OpenClaw Mako Hitlist -> YouTube Raw Metadata -> Manual Review
Product Concept + Stage Scope + Technical Plan (v2)

1. Executive concept

This program is a semi-automatic music data ingestion and manual curation system built around the official Israeli music chart at:
https://hitlist.mako.co.il/

At the current stage, the system is designed to collect chart entries from Mako Hitlist, extract only YouTube links, fetch raw YouTube metadata, and route records into a manual operator review workflow.

The system is intentionally not trying to solve full automatic lyrics extraction yet. Instead, it focuses on building a stable, production-oriented ingestion pipeline and a reliable operator workflow for manual finalization of song records.

The main purpose of this stage is to create a trustworthy operational foundation:
- stable chart ingestion
- deterministic YouTube metadata capture
- operator-facing review queue
- clean separation between raw and final data
- auditability
- future readiness for automation

2. Current stage objective

The goal of this stage is to implement a production-ready metadata ingestion + manual curation workflow.

The system must:
- ingest rows from the Mako Hitlist chart
- parse chart position, raw artist, raw song title, and outbound links
- keep only the YouTube link if present
- fetch and store raw YouTube metadata
- create a manual review task for the operator
- allow the operator to manually finalize the record:
  - final artist
  - final song title
  - final lyrics text
- persist both raw data and final reviewed data
- maintain audit trail and deduplication

3. Why this architecture is chosen now

This architecture is chosen intentionally for the first production stage.

Automatic lyrics extraction is currently the most fragile part of the possible pipeline. It is highly dependent on inconsistent YouTube descriptions, heuristic cleanup quality, and edge-case handling. Introducing that layer too early would increase complexity, reduce determinism, and make the MVP harder to validate and operate.

A raw metadata + manual review design is the fastest and safest path to a stable MVP because it:
- minimizes false positives
- removes the most brittle NLP/heuristic layer
- gives the operator full control over final data quality
- creates a validated corpus of real raw descriptions and reviewed outputs
- prepares the system for future partial automation based on real production data

At this stage, operational reliability and auditability are more important than maximum automation.

4. Operator workflow

The expected operator workflow is:

1. The system ingests chart rows from Mako Hitlist.
2. The system extracts only the YouTube link from each row, if present.
3. The system fetches raw YouTube metadata:
   - video title
   - channel title
   - description raw
   - published_at when available
4. The system creates a manual review task.
5. The operator opens the review task.
6. The operator reviews:
   - chart position
   - raw artist/title
   - YouTube URL
   - YouTube title/channel
   - raw YouTube description
7. The operator manually enters or corrects:
   - final artist
   - final song title
   - final lyrics text
8. The operator chooses one final decision:
   - approve
   - approve with edits
   - reject
   - no useful text

5. In-scope for this stage

Included in this stage:
- Mako Hitlist ingestion
- chart row parsing
- extraction of outbound links
- selection of YouTube link only
- YouTube URL normalization
- youtube_video_id extraction
- raw YouTube metadata fetch
- storage of raw chart and YouTube metadata
- manual review queue

- audit trail

- deduplication by youtube_video_id
- smoke checks and automated tests

6. Out-of-scope for this stage

Not included in this stage:
- automatic lyrics extraction
- LLM-based lyrics cleanup
- subtitle or captions parsing
- Spotify ingestion
- Apple Music ingestion
- publication pipeline
- auto-approval
- confidence scoring for lyrics extraction
- cross-platform music metadata aggregation
- recommendation or ranking logic
- downstream export/publishing workflows

7. MVP success criteria

The MVP is considered successful when:
- Mako chart ingestion works reliably end-to-end
- only the YouTube URL is selected from each chart row
- raw YouTube metadata is stored correctly
- a manual review task is created automatically
- the operator can complete the review from the UI
- raw data and final reviewed data are stored separately
- audit trail exists for key transitions and review actions
- deduplication works by youtube_video_id
- the system runs without manual database intervention for normal cases
- the system is ready for future automation layers without schema redesign

8. Architectural decision

At the current stage, the system is implemented as a metadata ingestion + manual curation pipeline:
OpenClaw automatically collects chart entries and raw YouTube metadata, while the operator manually creates the final song text and approves the final record.

9. Repo audit findings

- The target scope at this stage is intentionally narrowed:
  - no automatic lyrics extraction
  - no LLM cleanup
  - no auto-approval
- OpenClaw is used as the orchestration layer:
  - pipeline execution

  - https://hitlist.mako.co.il/

  - chart position

14. SQL schema v1

BEGIN;

CREATE TABLE chart_snapshot (
    id BIGSERIAL PRIMARY KEY,
    source_name TEXT NOT NULL,
    source_url TEXT NOT NULL,
    snapshot_date DATE NOT NULL,
    fetched_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    status TEXT NOT NULL,
    raw_payload_json JSONB,
    scraper_version TEXT,
    notes TEXT
);

CREATE TABLE chart_entry (
    id BIGSERIAL PRIMARY KEY,
    snapshot_id BIGINT NOT NULL REFERENCES chart_snapshot(id) ON DELETE CASCADE,
    chart_position INTEGER NOT NULL,
    artist_raw TEXT,
    song_title_raw TEXT,
    artist_norm TEXT,
    song_title_norm TEXT,
    youtube_url TEXT,
    youtube_video_id TEXT,
    has_youtube BOOLEAN NOT NULL DEFAULT FALSE,
    pipeline_status TEXT NOT NULL DEFAULT 'discovered',
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_chart_entry_snapshot_id ON chart_entry(snapshot_id);
CREATE INDEX idx_chart_entry_youtube_video_id ON chart_entry(youtube_video_id);
CREATE INDEX idx_chart_entry_pipeline_status ON chart_entry(pipeline_status);

CREATE TABLE youtube_video (
    id BIGSERIAL PRIMARY KEY,
    youtube_video_id TEXT NOT NULL UNIQUE,
    canonical_url TEXT NOT NULL,
    video_title TEXT,
    channel_title TEXT,
    description_raw TEXT,
    published_at TIMESTAMPTZ,
    metadata_fetched_at TIMESTAMPTZ,
    fetch_status TEXT NOT NULL,
    fetch_error TEXT,
    fetcher_version TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_youtube_video_fetch_status ON youtube_video(fetch_status);

CREATE TABLE review_task (
    id BIGSERIAL PRIMARY KEY,
    chart_entry_id BIGINT NOT NULL REFERENCES chart_entry(id) ON DELETE CASCADE,
    youtube_video_id_ref TEXT NOT NULL REFERENCES youtube_video(youtube_video_id) ON DELETE CASCADE,
    review_status TEXT NOT NULL DEFAULT 'pending',
    assigned_to TEXT,
    priority INTEGER NOT NULL DEFAULT 100,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    started_at TIMESTAMPTZ,
    completed_at TIMESTAMPTZ
);

CREATE INDEX idx_review_task_status ON review_task(review_status);
CREATE INDEX idx_review_task_chart_entry_id ON review_task(chart_entry_id);

CREATE TABLE review_result (
    id BIGSERIAL PRIMARY KEY,
    review_task_id BIGINT NOT NULL UNIQUE REFERENCES review_task(id) ON DELETE CASCADE,
    operator_id TEXT,
    final_artist TEXT,
    final_song_title TEXT,
    final_lyrics_text TEXT,
    decision TEXT NOT NULL,
    review_notes TEXT,
    reviewed_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_review_result_decision ON review_result(decision);

CREATE TABLE audit_event (
    id BIGSERIAL PRIMARY KEY,
    entity_type TEXT NOT NULL,
    entity_id TEXT NOT NULL,
    event_type TEXT NOT NULL,
    event_payload_json JSONB,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    actor_type TEXT,
    actor_id TEXT
);

CREATE INDEX idx_audit_event_entity ON audit_event(entity_type, entity_id);
CREATE INDEX idx_audit_event_event_type ON audit_event(event_type);

COMMIT;

16. OpenClaw workflow

16.1. Workflow name
mako_hitlist_youtube_raw_ingest_v1

16.2. Trigger
- manual run
- scheduled daily run

16.3. Stages

Stage A - Fetch chart page
Input:
- source URL

Action:
- open Mako hitlist
- wait for the list to load
- extract all visible chart items

Output:
- array of raw items

Stage B - Parse entries
Action:
- extract:
  - chart_position
  - artist_raw
  - song_title_raw
  - outbound links

Output:
- normalized raw chart entries

Stage C - Select only YouTube
Action:
- keep only YouTube from outbound links
- ignore Spotify and Apple Music
- if no YouTube exists, set status youtube_missing

Stage D - Canonicalize YouTube URL
Action:
- transform URL to canonical form
- extract youtube_video_id

Stage E - Fetch metadata
Action:
- call backend endpoint
- store video_title, channel_title, description_raw, published_at

Stage F - Create review task
Action:
- if metadata was fetched successfully:
  - create review_task
  - set status ready_for_manual_review

Stage G - Audit/logging
Action:
- write an audit event for each transition

18. Business rules

18.1. Deduplication
Canonical key:
- youtube_video_id

If the same video appears in different snapshots:
- youtube_video is not duplicated
- new chart_entry records are created
- a review task can either:
  - be created for each snapshot
  - or reuse a previously approved result

For MVP, recommended behavior:
- do not create a new review task if this youtube_video_id already has an approved or approved_with_edits result
- still save the new chart_entry

18.2. Retry policy
Retryable:
- timeout
- network error
- transient provider failure

Non-retryable:
- no YouTube URL
- invalid YouTube URL
- missing video ID
- 404-like permanent metadata failure

18.3. Review ownership
When Start Review is clicked:
- status -> in_manual_review
- assigned_to is recorded
- protect against parallel double editing

18.4. Final decisions
- approved
- approved_with_edits
- rejected
- no_useful_text
